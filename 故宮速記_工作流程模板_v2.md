# 故宮速記 → 導覽網站 工作流程 v2

## 用途定位

這不是學術圖錄。這是用戶的**美學速查系統**——拍的都是「覺得很酷」的東西，目的是跨域聯想和快速抓路徑。精度要求：大分類正確即可，細節後續上修。

---

## 給 Code 的指令

用戶會丟一批故宮拍的照片（本地路徑），分四種類型。
幫他 OCR 文字、整理成 MD 檔、最後編譯成可瀏覽的 HTML 導覽網站。

**資料來源：本地檔案優先。** 不走 Google Drive MCP，速度差太多。雲端只當備份。

---

## 照片分類規則（用戶自己分或 AI 判斷）

| 類型 | 內容 | 檔名前綴 |
|------|------|---------|
| 展品照 | 文物本體特寫 | `artifact_` |
| 告示牌 | 展品旁的小標籤（名稱、年代、編號） | `label_` |
| 介紹牌 | 展區牆面的大段說明文 | `wall_` |
| 視覺原理 | 展覽設計的圖解、原理說明 | `principle_` |
| ⏸️ 影片截圖待補 | 告示牌或介紹在影片中，尚未截圖 | `pending_` |
| ❌ OCR 失敗 | 太暗/模糊/手機畫質不足，無法辨識文字 | `failed_` |

---

## 配對邏輯（給 Code）

AI **不需要上網校對**。配對靠以下規則：

1. **拍攝順序推測**：連續編號的照片大概率屬於同一件展品（例如 IMG_4940 展品照 + IMG_4941 告示牌）
2. **OCR 名稱匹配**：告示牌 OCR 出的展品名稱 → 作為該組的 artifact_name
3. **配不上的 → 丟「未配對」區**：生成一個 `_unmatched.md`，列出所有無法自動配對的照片，用戶手動處理
4. **影片來源 → 丟「待補」佇列**：用戶標記 `pending_` 的項目先建空殼卡片，標註「待影片截圖補充」，後續用戶截圖後跑第二輪填入
5. **OCR 失敗分級處理**：
   - OCR 能讀 → 直接用
   - OCR 部分可讀（抓到材質/朝代/片段字） → 拿關鍵字查故宮線上典藏，限一次搜尋，有就補，沒有就標 `status: partial`
   - OCR 完全失敗 → 不查，直接丟 `_ocr_failed.md`

## 處理流程（給 Code）

```
1. 用戶丟一批本地照片（資料夾路徑）
2. AI 判斷每張照片類型（展品/告示牌/介紹牌/原理/pending）
3. OCR 所有含文字的照片
4. 配對邏輯：
   a. 按拍攝順序推測同組
   b. OCR 名稱匹配確認
   c. 配不上 → _unmatched.md
   d. pending_ → 建空殼卡片，status: pending_video
   e. OCR 部分可讀 → 查故宮線上典藏補全（限一次），補到就 complete，補不到就 partial
   f. OCR 完全失敗 → _ocr_failed.md，記錄檔名+失敗原因
5. 判斷：零散小件群 → 群組卡片 / 獨立展品 → 單件卡片
6. 按模板生成 MD 檔
7. 生成展區總覽 MD
8. 編譯成 HTML 導覽網站（靜態）
```

## HTML 導覽網站規格

- 單頁或少頁，側邊欄導航
- 按展覽 → 展區 → 展品/群組 三層結構
- 每件展品：照片 + 基本資訊 + 描述
- 群組卡片：多張照片 + 概述
- 待補項目顯示 ⏸️ 標記，部分辨識顯示 🟡 標記，OCR 失敗項目顯示 ❌ 標記，都不隱藏
- 搜尋或篩選功能（按朝代、材質、類型）
- 視覺風格：深色底、展品照突出、字體乾淨
- 純靜態 HTML，不需要 server

---

## 用戶操作 SOP

### 第一輪（照片）
1. **拍照時**：每件展品拍 2-3 張（本體 + 告示牌），介紹牆單獨拍
2. **回家後**：照片放本地資料夾，丟給 Code，附一句「故宮 [展覽名]，幫我整理」
3. **Code 處理**：分類 → OCR → 配對 → 生成 MD → 編譯 HTML
4. **用戶檢查**：看 `_unmatched.md`，手動配對或捨棄
5. **快速複習**：開 HTML 點點點，15 秒一件快速掃

### 第二輪（影片截圖，時間不限）
1. 從影片中截取告示牌/介紹牌畫面
2. 命名為 `pending_XXX` 對應的檔名
3. 再丟一次 Code：「補充截圖，更新 pending 項目」
4. Code 填入空殼卡片，status 改 complete
